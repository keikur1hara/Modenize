# AGENTS.md — リバースエンジニアリング用 Codex指示書

このファイルを既存VB.NETリポジトリのルートに配置してください。
Codexはこのファイルを読み、タスク管理表（tasks.md）に従って設計ドキュメントを自律的に生成します。

---

## あなた（実行者）の操作

Codexに以下のように指示するだけで進められます。

| 指示 | Codexの動作 |
|---|---|
| `次` | tasks.md を読み、次の未着手タスクを実行中にして作業を開始する |
| `完了` | 現在のタスクを完了にし、次の実行可能タスクを提示する |
| `状況` | tasks.md の進捗サマリーを表示する |
| `やり直し T-XX` | 指定タスクを未着手に戻し、成果物を再生成する |
| `T-XX` | 指定タスクを実行する |

---

## Codexへの共通ルール

### 基本動作

1. タスクを開始するとき、tasks.md のステータスを `実行中` に更新する
2. 成果物を `docs/` 配下の指定パスにMarkdownファイルとして出力する
3. タスク完了時、tasks.md のステータスを `完了` に更新する
4. 依存タスクが未完了のタスクは実行しない（依存関係は tasks.md に記載）

### 出力ルール

- 日本語で記述すること
- Markdown形式で出力すること
- 図表は Mermaid記法 を使用すること
- ソースコードから読み取れる「事実」と「推測」を区別すること
  - 事実: そのまま記述
  - 推測: 【推測】を先頭に付与
- 不明点には【要確認】を付けること
- マジックナンバーやハードコードされた値にも【要確認】を付けること
- 各ドキュメントの先頭に「対象範囲」と「解析対象ファイル数」を記載すること

### コンテキスト管理

- `*.Designer.vb` は原則除外（コントロール名一覧が必要な場合のみ参照）
- `*.resx`, `bin/`, `obj/`, `packages/` は除外
- 1タスクの解析スコープが大きすぎる場合、プロジェクト単位で分割して処理すること
- 前タスクの成果物を参照する場合、`docs/` 配下の該当ファイルを読むこと

## タスク定義

各タスクのIDと実行内容を以下に定義する。
進捗管理は `tasks.md` で行い、このファイルは変更しない。

### T-00: リポジトリ規模調査

**出力先:** `docs/00_survey.md`

以下を調査してテーブル形式で出力:
- プロジェクト別ファイル数（プロジェクト名 / .vbファイル数 / .Designer.vbファイル数 / その他）
- 合計ファイル数・合計行数（概算）
- プロジェクト規模ランク（Large / Medium / Small）
- Form / UserControl の総数

完了後、画面数・機能規模に応じて tasks.md の T-31〜 T-41〜 の行を追加・調整すること。

### T-10: 全体構造の把握

**依存:** T-00 完了後
**出力先:** `docs/01_system_overview/`

以下のファイルを生成:

**solution_structure.md:**
- .slnファイルに含まれるプロジェクト一覧（プロジェクト名 / 種別 / ターゲットFW / 出力タイプ）
- プロジェクト間の参照関係（Mermaidグラフ）

**architecture.md:**
- レイヤー構成（UI層 / ビジネスロジック層 / データアクセス層 / 共通層）
- 各レイヤーに属するプロジェクト・名前空間のマッピングテーブル
- アーキテクチャ図（Mermaid）

**tech_stack.md:**
- .NET Frameworkバージョン
- NuGetパッケージ一覧（packages.config / PackageReference から。名前 / バージョン / 用途推定）
- 外部ライブラリ・COM参照一覧
- 使用フレームワーク（WCF / WinForms / Crystal Reports等）

**startup_flow.md:**
- Main関数の場所と処理
- 起動時の初期化処理
- 最初の画面とログインフローの有無
- 起動フロー（Mermaidシーケンス図）

### T-31〜: 個別画面仕様書

**依存:** T-30 完了後
**参照:** `docs/03_screens/screen_list.md`
**出力先:** `docs/03_screens/screens/{フォーム名}.md`

T-00の規模調査結果に基づき、以下のルールでバッチ分割:
- 小規模（〜20画面）: T-31 に全画面
- 中規模（20〜50画面）: T-31〜T-35 に5〜10画面ずつ
- 大規模（50画面〜）: T-31〜 にプロジェクト単位または3〜5画面ずつ

各画面について:
1. 画面名・タイトル・概要
2. 配置コントロール一覧（コントロール名 / 種別 / 表示テキスト / 主要プロパティ）
3. 初期表示処理（Form_Load等）
4. イベントハンドラ一覧（コントロール名 / イベント / 処理概要）
5. 各イベントハンドラの詳細処理フロー
6. バリデーションルール一覧（対象項目 / ルール / エラーメッセージ）
7. 使用テーブル・ストアドプロシージャ
8. 呼び出し元画面・遷移先画面
9. エラーメッセージ一覧（MsgBox / MessageBox.Showから抽出）

### T-R0: クロスチェック

**依存:** T-20 AND T-31〜 AND T-41〜 AND T-50 全完了後
**出力先:** `docs/review/cross_check.md`

以下の観点でドキュメント間の整合性を確認:
- Phase 3 画面一覧 × Phase 2 CRUD表: 全画面がCRUD表に存在するか
- Phase 2 CRUD表 × Phase 3 画面一覧: CRUD表の画面名が画面一覧と一致するか
- Phase 4 機能一覧 × Phase 3 画面一覧: 機能が画面を網羅しているか
- Phase 5 認証等 × Phase 2 テーブル一覧: 非機能で使うテーブルがDB設計に含まれるか

不整合があれば一覧テーブルで報告。

### T-R1: 推測・要確認の棚卸し

**依存:** T-R0 と同時実行可
**出力先:** `docs/review/open_questions.md`

`docs/` 配下の全ドキュメントから【推測】【要確認】を抽出し、一覧化:
- マーカー種別（推測 / 要確認）
- 記載ドキュメント名
- 該当箇所の内容
- 確認方法の提案
